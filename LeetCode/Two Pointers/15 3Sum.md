## 1. Problem (in my own words)
[[1 Two Sum]], but three elements, and they have to add up to `0`. Plus, instead of returning the indices, you just have to return the values.  However, you need to return all of the possible solutions, and no solutions might happen, too. All the value triplets (the triplets, not the individual values in the triplet) should be unique, and the order doesn't matter.

## 2. Constraints & what they imply
[[Constraint to Complexity Reference|See here for reference]]

| Constraint                 | Implication for approach                                                                                                               |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `3 <= nums.length <= 3000` | up to $O(n^2 \cdot log n)$<br>sorting is free game, baby. Plus, it asks for values instead of indices, so all the more reason to sort. |
| `-10^5 <= nums[i] <= 10^5` | int32 is enuff.                                                                                                                        |

## 3. Recognition trigger
It's awfully similar to [[1 Two Sum|two sum]]... and guess who just learned [[167 Two Sum II - Input Array is Sorted|Two Sum but sorted]]....

## 4. Brute force

> [!info]- Idea
> Find all combinations for triplets and check if they sum to zero.
---
>[!info]- Complexity 
>time $O(n^3)$ / space $O(1)$
---
> [!warning]- Why it's not enough?
> Because, you're doing redundant work. It's better if you structure it in such a way where redundancy is as minimal as possible.
## 5. Optimal approach

> [!info]- Complexity 
>time $O(n^2)$ / space $O(1)$
---
> [!info]- Idea
> Sort first, and then iterate through the array and use [[167 Two Sum II - Input Array is Sorted|Two Sum II]], with the target being `-nums[i]` .
---
> [!info]- Why it works (the key insight)
> Because. You're only inspecting each triplet once, and you know how to navigate and narrow the search space for the target efficiently. 

## 6. Code
```python
# language: python
class Solution:
    def twoSum(self, nums: list[int], l: int, target: int) -> list[int]:
        r = len(nums) - 1
        ans = []
        while l < r:
            res = nums[l] + nums[r]
            if res == target:
                ans.append([nums[l], nums[r], -1*target])
                l += 1
                r -= 1
                while l < r and nums[l] == nums[l-1]: # skip over same values to prevent duplicates
                    l += 1
                while l < r and nums[r+1] == nums[r]: # skip over same values to prevent duplicates
                    r -= 1
            elif res < target:
                l += 1
            else:
                r -= 1
        
        return ans

    def threeSum(self, nums: list[int]) -> list[list[int]]:
        nums.sort()
        length = len(nums)
        r = length - 1
        ans = []

        for i in range(length):
            if i > 0 and nums[i-1] == nums[i]: # optimization, skip over the same values
                continue

            res = self.twoSum(nums, i+1, -1*nums[i])
            ans.extend(res)
            
        return ans
```

## 7. Mistakes I actually made
- I instantly returned once the answer was found on the `twoSum` function, when it's still possible to get other solutions.
- Skip over the same values in finding other solutions in the `twoSum` function to prevent duplicates.
